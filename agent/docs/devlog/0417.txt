== 关键设计决策 ==

1. LLM 两次调用分工
   第一次：流式 agentic loop，tool call 循环，结果直接推 SSE
   第二次：非流式 JSONMode，把原始回答提炼为 StructuredResponse
   理由：风格层不直接面对散文，避免改坏信息

2. 风格层永远是 post-processor
   不参与工具调用，不补充知识，只改句式/语气/节奏
   失败时 fallback 到 StructuredResponse.FinalAnswer，主链路不受影响

3. 风格路由
   high risk → plain（强制）
   chat/reflection/discussion → sharp
   tutorial/coding → blunt
   emotional/risky → plain
   default → blunt

4. 可观测性从第一天接入
   每个 turn 记录：first_token_latency / tool_latency / style_latency / total_latency
   fallback_reason 和 timeout_stage 显式字段，不靠"感觉"判断问题出在哪

== 遇到的问题 ==

- go-openai v1.24 中 ToolCallDelta.Index 是 *int 而非 int，需要 nil 判断
- go-openai 的 FunctionDefinition.Parameters 类型是 any，直接传 json.RawMessage 即可，不需要先 Unmarshal 成 openai.Definition（该类型不存在）
- tool.Handler 签名用 json.RawMessage，executor 和 main 的调用点要对齐类型

## 新增真实工具实现

internal/tool/shell.go
- NewShellExecTool: 执行 bash -c 命令，stdout+stderr 合并返回
- 非零退出码包含在输出里而非作为 error，让 agent 可读取失败信息

internal/tool/fileread.go
- NewFileReadTool: 读取文件内容，64KB 截断并附说明

internal/tool/search.go
- NewWebSearchTool: 调用 Tavily API 搜索，返回摘要 + top N 结果
- API key 无配置时返回明确错误，不在启动时 panic
- 支持 MaxResults 配置

cmd/server/main.go
- registerBuiltinTools 改为接收 *Config，注册 shell_exec / file_read / web_search
- Search API key 支持 GO_AGENT_SEARCH_API_KEY 环境变量覆盖
- Config 新增 Tool.WorkDir、Tool.BaseDir、Search.APIKey、Search.MaxResults 字段
- EngineConfig 传入 MaxTokens 和 Temperature（原来硬编码）

configs/config.yaml
- tool 段新增 work_dir、base_dir 字段
- 新增 search 段（api_key、max_results）
