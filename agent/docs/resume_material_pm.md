## 一、项目背景与问题定义

### 核心问题陈述

在长期使用主流 AI Agent 框架（OpenClaw 类）过程中，识别出三类系统性体验问题：

1. **运行时不透明**：网络连接、流式输出、工具执行状态无法观测，出现问题难以定位是模型、网络还是框架本身
2. **风格控制失效**：基于 Prompt 的人格约束在长对话、工具调用后持续漂移，无法稳定维持特定回答风格
3. **"AI 客服假笑"问题**：大模型默认社交策略过于礼貌，先安慰后判断，产生大量无信息量输出

### 问题量化

- 现有框架中风格 Prompt 在超过 5 轮对话后漂移率显著上升
- 工具调用结果无法追踪，链路黑箱导致调试成本高
- 主观体验：回答有帮助感，但缺乏判断感和直接感

## 二、产品决策与设计思路

### 关键决策一：Runtime 稳定性优先于风格表达

**决策背景**：项目初期面临两个方向——直接做"更有个性的 AI"，还是先重构底层运行时。

**决策结果**：确立"先换底盘，再谈风格"的分阶段原则，避免在不稳定的 Runtime 上堆砌风格功能。

**可用于简历的措辞**：
- 主导确立产品优先级框架，将 Runtime 稳定性设为 Phase 1 核心目标，风格层作为 Phase 2 接口预留，避免架构返工
- 识别"在不稳定底层上叠加功能"的风险，推动团队先完成基础设施再做体验创新

### 关键决策二：内容生成与风格表达解耦

**决策背景**：传统方案是在 System Prompt 里同时写任务指令和风格要求，导致两者互相干扰。

**决策结果**：设计双模型架构——大模型负责推理和工具调用，独立的风格后处理层负责改写输出语气，两者之间通过结构化中间对象（StructuredResponse）传递，明确划定不可改写字段。

**可用于简历的措辞**：
- 设计双模型解耦架构，将内容生成与风格表达分离为独立处理层，通过结构化中间对象传递，降低风格改写破坏原始信息的风险
- 定义可改写项（语气、句式）与不可改写项（数字、步骤、风险提示）的明确边界，将"只改语气不改信息"从口头原则落为系统约束

### 关键决策三：显式可观测性而非主观体感优化

**决策背景**：此前的体验优化停留在"感觉更快了"的主观判断，缺乏可量化依据。

**决策结果**：将可观测性列为 v0.1 必须具备的能力，要求系统在每轮对话中记录首 token 延迟、工具调用耗时、风格层耗时、回退原因等指标。

**可用于简历的措辞**：
- 推动将可观测性纳入 MVP 交付范围，建立包含首 token 延迟、工具耗时、风格层 overhead 的指标体系，将"感觉更快"转化为可对比的数据结论
- 设计 Fallback 追踪机制，每次风格层降级均记录原因，为后续迭代提供决策依据

## 三、范围管理与优先级

### MVP 范围划定

明确定义 v0.1 的"做"与"不做"：

**做（Phase 1）**：
- 稳定的单会话 Agent 流程
- 多 LLM Provider 支持（OpenAI、Anthropic、Ollama、OpenAI-compat）
- 流式输出 + 工具并发执行
- 结构化中间层 + 风格处理接口预留
- 基础指标采集与日志

**不做（明确排除）**：
- 多 Agent 图编排
- 完整长期记忆系统
- LoRA 训练闭环
- 复杂 Dashboard
- OpenClaw 全功能兼容层

**可用于简历的措辞**：
- 制定清晰的 MVP 范围，明确排除多 Agent 编排、完整记忆系统等高复杂度需求，聚焦核心链路，确保 Phase 1 可交付
- 输出结构化的差距分析文档（GAP Analysis），将系统问题按优先级分为"阻塞稳定性"、"影响生产可用性"、"v0.2+ 范围"三级，指导开发顺序

### 迭代节奏

在单次开发周期（0420）内完成 9 个独立补丁，包含：
- 工具并发执行改造（goroutine + WaitGroup）
- MustKeep 内容校验机制
- 结构化响应降级兜底
- /metrics HTTP 端点
- LLM 指数退避重试
- 滚动摘要记忆
- BoltDB Session 持久化
- 并发 Turn 竞争修复
- 类型化 SSE 工具事件流

## 四、生态与扩展性设计

### ClawHub Skill 集成

**背景**：ClawHub 是 OpenClaw 生态的 Skill 注册中心，拥有 50,000+ 社区 Skill，但与目标 Runtime 架构不兼容。

**方案**：设计 Skill Loader，兼容 ClawHub 的 SKILL.md 格式（YAML frontmatter + markdown body），无需运行 OpenClaw 本体即可复用社区资产。

**效果**：用户只需将 Skill 目录放入 `skills/`，服务器启动时自动解析注册，零代码接入。

**可用于简历的措辞**：
- 设计第三方 Skill 生态接入方案，兼容 ClawHub（50,000+ Skills）标准格式，实现社区资产零代码复用，降低用户使用门槛
- 通过格式抽象屏蔽上游生态依赖，用户无需安装或运行 OpenClaw 即可消费其生态资源

### 用户体验细节

- 分层配置设计：项目配置（可提交 git）与用户凭证（`~/.config/go-agent/`）分离，API Key 永不进入代码仓库
- `make init-config` 一键初始化用户配置，降低首次使用门槛
- 服务启动失败提供明确的 warn 日志而非 panic，保证可调试性

## 五、技术深度体现（PM 视角）

以下内容体现对技术实现的理解深度，适合在面试中展开：

| 技术概念 | PM 理解角度 |
|---------|------------|
| SSE 流式输出 | 首 token 延迟是用户感知响应速度的关键指标，非总响应时间 |
| 工具并发执行 | 同一轮多工具调用从串行改为并发，wall time 从 N×T 降为 T，直接影响 P90 延迟 |
| 结构化中间层 | 将散文输出转为结构化对象，是内容保真与风格改写解耦的工程化手段 |
| 指数退避重试 | 区分可重试错误（429、5xx）与不可重试错误（4xx），避免无效重试放大下游压力 |
| 滚动摘要记忆 | 解决 LLM Context Window 限制，通过压缩旧对话保留语义同时控制 Token 成本 |
| Fallback 机制 | 风格层作为非必要组件，失败时静默降级到原文输出，保证核心功能可用性不受影响 |
